# 携程数据采集阶段性汇报

> 2026-06-22 | 状态：核心功能完成，待扩展

---

## 一、选择转向

### 从"OSM+携程双体系"转向"携程ID体系为主"

| 项目 | 旧方案 | 新方案 | 原因 |
|------|--------|--------|------|
| 景点ID体系 | OSM osm_id + 携程虚拟ID | 直接用携程 poi_id | 简化系统复杂度 |
| 数据来源 | OSM + 携程 + 高德 | 携程为主 | 携程数据更丰富、坐标更准确 |
| OSM角色 | 核心数据源 | 仅冷启动 | 找小众景点时才用 |
| 入库逻辑 | 匹配OSM再入库 | 直接入库 | 携程100%有坐标，不需要匹配 |

### 决策原因

1. **携程数据本身有完整坐标**：lat, lng 字段100%有值，不需要OSM坐标
2. **携程有丰富字段**：名称、评分、评论数、标签、封面图、开园状态
3. **OSM匹配率太低**：只有16%，84%的携程景点无法入库
4. **系统复杂度应该最小化**：不需要两套ID体系混在一起

### 影响

- ❌ 不再需要OSM匹配逻辑
- ❌ 不再需要用 `ctrip_{poi_id}` 虚拟ID（但为兼容保留）
- ✅ 直接用携程 `poi_id` 作为主键
- ✅ 所有携程景点都能入库（100%）
- ✅ OSM仅在冷启动时使用（找小众景点）

## 三、技术实现

### 3.1 采集流程

```
携程API → collect_ctrip.py → ctrip_attractions.json
                                    ↓
                            import_ctrip_to_db.py
                                    ↓
                            poi_details + poi_images
```

**关键特性：**
- REST API 批量采集，不需要 Playwright
- 限速 10-15 秒/请求，防 WAF 封号
- 全局 poi_id 去重，防止重复采集
- 噪音过滤：演出、酒店、餐厅等非景点数据

### 3.2 后端 API

**GET /api/pois**

查询携程景点数据

**参数：**
- `category`：携程分类标签（如"历史建筑"）
- `city`：城市名称
- `limit`：数量限制（默认25）

**返回：**
```json
{
  "pois": [...],
  "total": 25,
  "category": "历史建筑",
  "categories": ["历史建筑", "自然山水", ...]
}
```

### 3.3 前端集成

- 创建 `ctripPoiService.ts` - 调用后端 API
- 更新 `tripStore.ts` - 添加携程景点状态管理
- 更新 `MapPanel.vue` - 添加分类选择器 UI

**使用方法：**
1. 在地图左上角找到**罗盘图标**（携程景点分类按钮）
2. 点击展开分类面板
3. 选择一个分类（如"历史建筑"）
4. 地图上会显示该分类的景点（最多25个）

## 四、坐标准确性验证

### 4.1 验证方法

### 4.2 验证结果

| 故宫博物院 | 39.924091, 116.403414 | 39.917431, 116.390782 | < 1km |
| 洪崖洞 | 29.568345, 106.58542 | - | 市区范围内 |
| 解放碑 | 29.563463, 106.581786 | - | 市区范围内 |
