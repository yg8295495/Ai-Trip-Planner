## 1. 项目定位与核心产品哲学

本系统定位为**极其客观的“三维时空数字化账本”**。系统放弃传统的“由 AI 代替用户全权决定路线”的执法者思维，将自驾行程的试错权、审美偏好和策略路线选择权**完全归还给用户**。

* **双向规划流（直觉对齐）**：
1. **自上而下**：输入大起终点 $\rightarrow$ 生成宏观长线 $\rightarrow$ 通过过滤罗盘标记地标 $\rightarrow$ 动态切片分天 $\rightarrow$ 下钻单日雷达精修（酒店、餐饮、露营小溪）。
2. **自下而上**：用户自带强烈预期 $\rightarrow$ 自由在地图或列表中打点 $\rightarrow$ 标记每日停靠与过夜点 $\rightarrow$ 系统逆向缝合汇总 $\rightarrow$ 最终推演生成总览长线。


* **数字账本原则**：每一次景点的增删、每一次过夜点的勾选，系统只在底层一维状态树上做纯粹的数学累加与路网局部缝合，不阻塞、不自作主张修改用户意图。

## 2. 系统核心技术栈与边界规范

### 2.1 前端核心骨架

* **核心框架**：Vue 3 (Composition API / `<script setup>`) + Vite。
* **样式与UI系统**：Tailwind CSS + `shadcn-vue`（风格：New-York，字体：Outfit，图标库：`lucide-vue` / `@iconify/vue`）。
* **动效与交互组件**：
* `@formkit/auto-animate`：负责 DOM 节点（如珍珠链、标签墙）增删切换时的丝滑物理动画。
* `vue-draggable-plus`：负责一维珍珠项链、跨天/天内节点的物理拖拽排序。


* **地图底座**：高德地图 JS API v2.0（禁止任何第三方未经封装的 DOM 污染）。

### 2.2 后端与性能边界

* **核心框架**：FastAPI (Python 3.10+)。
* **状态性质**：**彻底的无状态（Stateless）API 代理**。现阶段暂缓引入任何复杂的外部数据库，以最小化系统复杂度。
* **数据检索机制**：自驾长廊检索等消耗算力的空间几何匹配，全部依靠应用启动时一次性常驻在后端内存中的 `ctrip_attractions.json` 字典。利用 Python 的 `Set` 和 $O(1)$ 级哈希字典进行秒级 adcode/经纬度长廊裁剪。

## 3. 数据架构与核心对象字段定义 (Frozen)

数据结构分为三大核心对象：**POI源数据**、**珍珠链状态树（Waypoint）**、**高德路由几何数据**。

### 3.1 携程 POI 主键源数据模型 (`candidatePois`)

全盘摒弃原“OSM+携程双轨虚拟 ID 体系”，全面固化为以携程真实数据为主键、OSM为微观冷启动长尾的单轨结构。

```typescript
interface CtripPoi {
  poi_id: string;          // 主键，格式如 "ctrip_12345" 或 纯数字字符串
  name: string;            // 景点/POI 完整名称
  lat: number;             // 高德/火星坐标系纬度
  lng: number;             // 高德/火星坐标系经度
  keytag: string;          // 评级标签（"5A", "4A", "国家地质公园", "世界文化遗产" 或 空）
  score: number;           // 携程评分 (0.0 - 5.0)
  comment_count: number;   // 评论数
  category: string;        // 属于 13 类自驾过滤标签中的具体哪一类
  cover_url?: string;      // 封面图链接
  open_status?: string;    // 开园状态/营业时间
}

```

### 3.2 珍珠链途经点状态树模型 (`Waypoint`)

存储于 `routeStore.ts` 中的一维扁平数组。

```typescript
interface Waypoint {
  id: string;              // 前端生成的唯一唯一标识符（UUID / Timestamp），用于 Draggable 渲染
  poi_id: string;          // 关联的底层 POI 源数据主键
  poi_details: CtripPoi;   // POI 详情副本（只读）
  isOvernight: boolean;    // 【核心指针】是否在此处作为当天的终点休息过夜
  sequence: number;        // 在一维珍珠项链中的物理排序索引
}

```

### 3.3 高德路由分段大盘模型 (`routeInfo`)

存储于 `tripStore.ts` 中，由高德 API 返回的纯客观物理轨迹。

```typescript
interface RouteSegment {
  start_poi_id: string;
  end_poi_id: string;
  distance_meters: number; // 两点间物理距离
  duration_seconds: number;// 两点间驾车耗时（秒）
  polyline: [number, number][]; // 这一段的高德折线轨迹点集
}

```

## 4. 三维空间决策矩阵算法

系统根据用户的缩放层级与操作意图，在三大空间算法间进行无缝自动切换，坚决杜绝移动地图时盲目刷屏：

| 决策视角 | 触发条件 | 核心空间算法 | 数据源 | 过滤与视觉渲染策略 |
| --- | --- | --- | --- | --- |
| **1. 盲选周边** | 用户在无路线上进行地图任意打点 | **圆形缓冲区（Buffer）**<br>

<br>半径可选 5/10/20km | 携程本地库 | 按评分降序。地图展示经典聚合 Marker，默认不开启自动扫描。 |
| **2. 总览长廊** | 起终点生成，长线 Polyline 渲染完成 | **沿途多边形走廊（Corridor）**<br>

<br>两侧各 5km - 15km 缓冲区 | 内存字典字典匹配 | **5A/4A/世界遗产绝对优先**。13类标签罗盘初始全灭，用户主动点击哪类，地图才“闪现”对应分类 Marker。 |
| **3. 分日精修** | 用户点击下钻进入 Day $X$ 详情面板 | **单日行驶轨迹线多边形 / 停靠点周边** | 高德生态分类 + **OSM骨架库** | 触发微观雷达扫描动效。优先检索高德周边的酒店餐饮；若用户寻找野景，**冷启动 OSM 提取长尾小溪、野生露营点**。 |

## 5. “珍珠项链”时空重排交互模型

这是本规划器的灵魂，采用 **一个扁平状态树，派生出多维分天视图** 的高度响应式逻辑。

```
【一维扁平状态树 waypoints】
[ 起点 A ] → [ 景点 B (3h) ] → [ 景点 C (2h) (isOvernight: true) ] → [ 景点 D (5h) ] → [ 终点 E ]
                                     ↓（Pinia 计算属性自动打结切片）
【衍生出多维分天视图 routeByDays】
 * Day 1: [ 起点 A ] → [ 景点 B ] → [ 景点 C(宿) ]   (当日总车程：5小时)
 * Day 2: [ 景点 C(启) ] → [ 景点 D ] → [ 终点 E ]   (当日总车程：5小时)

```

### 5.1 时间动态塌陷与重排算法

* 当用户在一维列表的某个节点上点击勾选了 `isOvernight = true`，Pinia 中的 `computed` 属性 `routeByDays` 会瞬间监听到状态改变，立即在此处执行**物理切片断开**。
* 后续所有节点（如 景点D、终点E）的时间车耗自动向内塌陷，归集到 Day 2 的看板中。
* **删除节点的动态连线交互**：当用户删除“景点 B”时，前端直接销毁原 `A $\rightarrow$ B` 和 `B $\rightarrow$ C` 的折线，静默向高德请求一段全新的 `A $\rightarrow$ C` 轨迹并缝合。**地图呈现出路线像皮筋一样被扯动并自动弹回扣在 C 点的顺滑视效**。

### 5.2 手机端高德原生导航无缝导出（二维码方案）

网页规划完成后，为了在实地自驾中完美闭环，系统提供**分天高德原生 Schema 扫码唤起能力**。

* **URI 拼接契约**：
`amapuri://route/plan/?sid=起点ID&slat=...&did=终点ID&dlat=...&pts=途经点JSON&dev=0&m=0`
* **交互体验**：每一天的卡片（DayCard）右上角常驻一个“手机导航”图标。点击后，前端将当天拆分出来的起终点和最多 16 个途经点坐标拼接成上述 Schema，转为二维码弹窗。用户掏出手机微信/相机一扫，**直接唤起手机上的原生高德地图 App**，且行程顺序、停靠点完全一致，一键点击“开始导航”即可出发。

## 6. 大模型副驾驶（AI 军师）解耦与联动交互

大模型被彻底夺取写权限。它不再参与底层 JSON 的数据结构拼写，也不管坐标定位，而是作为**高智商的纯粹只读审阅者**。

### 6.1 初始化引导约束

大模型启动时通过 System Prompt 锁死格式，主动表白身份并下发快捷 Chips（如：`#带娃出行` `#看重秋色` `#硬核徒步`），引导用户锁定出行模式。

### 6.2 时空上下文注入与被动触发（军师提案机制）

* **运行逻辑**：整个过程中大模型绝不阻塞前端。每当用户生成路线或大规模调整景点时，前端会静默将当前行程快照（如：`月份: 10月`, `当前选定景点ID`, `长廊沿途未选的高星级景点`）打包成一个 JSON 发给大模型。
* **人情味提案**：大模型审阅这段 JSON 后，利用其宏观知识在左侧聊天面板中发言。
* *例如*：“主公！我发现您 10 月中旬规划了川西线，且路过了小金县，但你居然漏掉了‘四姑娘山双桥沟’！这时候正是它彩林最爆裂的季节，距离你 Day 1 住宿酒店仅 20 分钟车程，建议立刻加入。”



### 6.3 终极交互：【一键加入行程】

* 大模型在输出上述推荐文本时，后端解析器会识别到其推荐的 POI 唯一主键（`ctrip_999`）。
* 左侧 `ChatPanel` 的 Markdown 渲染器会捕获该标签，并在文本中该景点名称右侧，**动态渲染一个极具吸引力的原生 UI 按钮：【➕ 一键塞进 Day 1】**。
* 用户点击该按钮，直接触发前端事件总线，调用 `routeStore.addWaypoint()`。**大模型动了嘴，用户动了手，数据树瞬间重排，实现了惊艳的智能联动。**

### 7.1 面板核心组件受控职责列表

1. **`MapPanel.vue` (画布)**：不保留自主查询状态。只在路线生成时绘制大动脉折线；渲染标签罗盘亮起时的 Marker。InfoWindow 内部的加入按钮一律桥接至全局 `window.__addToItinerary`。

## 8. Agent 协作与代码合并最高执行法

任何本地 Agent 或研发人员在对本项目进行代码编写时，必须严格遵守以下执行序列：

1. **只做增量注入，不做毁灭重写**。在面对大体积历史屎山组件时，先通过注释标定“安全保护区”与“手术插入点”。
2. **确保一维扁平状态树（Pinia Store）先于 UI 视图建立**。任何界面的重排、分天，必须完全由 Store 中的 `isOvernight` 属性变化联动计算属性响应式派生，严禁在 `.vue` 组件内部私吞或缓存拼装临时天数数组。
3. **样式与交互必须使用已安装的基础设施**。过渡动效一律包裹 `v-auto-animate`，拖拽一律采用 `VueDraggable` 规范，UI 结构严格执行 `shadcn-vue` 规范。
